## ЁЯФД Flyway Migration History

PawFinder uses **Flyway** for versioned, repeatable database migrations. Each service that requires database tables has its own set of migrations in `src/main/resources/db/migration/`.

### Applied Migrations (V1 - V7)

| Version | Service | File | Description |
|---------|---------|------|-------------|
| V1 | Auth | `V1__create_users_table.sql` | Create `users` table (auth provider, display name, auth ID) |
| V1 | Alert | `V1__init.sql` | Enable `uuid-ossp` extension |
| V2 | Alert | `V2__create_pets_table.sql` | Create `pets` table (owner, species, breed, QR code) |
| V3 | Alert | `V3__create_alerts_table.sql` | Create `alerts` table (lost pet alerts with location, reward) |
| V4 | Alert | `V4__create_sightings_table.sql` | Create `sightings` table (user reports with location evidence) |
| V5 | Alert | `V5__add_postgis_location.sql` | Add PostGIS `GEOGRAPHY(Point)` to `alerts.last_seen_location` |
| V5 | Messaging | `V5__create_conversations_table.sql` | Create `conversations` table (owner-finder chat threads) |
| V6 | Alert | `V6__add_sighting_postgis_location.sql` | Add PostGIS `GEOGRAPHY(Point)` to `sightings.location` |
| V6 | Messaging | `V6__create_messages_table.sql` | Create `messages` table (chat messages with read status) |
| V7 | Auth | `V7__add_email_password.sql` | Add `email` and `password_hash` columns to `users` |
| V7 | Analytics | `V7__create_daily_metrics_table.sql` | Create `daily_metrics` table |

### Flyway Configuration (per service)

```yaml
# auth-service
spring:
  flyway:
    enabled: true
    locations: classpath:db/migration
    table: flyway_auth_history   # Custom history table to avoid conflicts
```

> **Note:** Each service has a separate Flyway history table (e.g., `flyway_auth_history`, `flyway_alert_history`) to prevent migration conflicts across services sharing the same database.

## ЁЯзн PostGIS Spatial Features

The database leverages **PostGIS** for location-based queries:

### Spatial Columns

| Table | Column | Type | Purpose |
|-------|--------|------|---------|
| `alerts` | `last_seen_location` | GEOGRAPHY(Point, 4326) | Pet's last known location |
| `sightings` | `location` | GEOGRAPHY(Point, 4326) | Sighting report location |

### Spatial Indexes

| Table | Index | Type | Use Case |
|-------|-------|------|----------|
| `alerts` | `idx_alerts_location_geog` | GIST | Proximity queries (`ST_DWithin`) |
| `sightings` | `idx_sightings_location_geog` | GIST | Proximity queries (`ST_DWithin`) |

### Example Queries (PostGIS)

```sql
-- Find alerts within 5km of a point
SELECT * FROM alerts
WHERE ST_DWithin(
  last_seen_location,
  ST_SetSRID(ST_MakePoint(-122.4194, 37.7749), 4326)::geography,
  5000
) AND status = 'active';

-- Find sightings near an alert's last known location
SELECT s.* FROM sightings s
JOIN alerts a ON s.alert_id = a.id
WHERE ST_DWithin(s.location, a.last_seen_location, 1000);

-- Spatial heatmap: count alerts per grid cell
SELECT
  ST_SnapToGrid(a.last_seen_location::geometry, 0.01) AS cell,
  COUNT(*) AS alert_count
FROM alerts a
WHERE a.status = 'active'
GROUP BY cell;
```

## ЁЯФМ Connection from Services

Services connect to the database via JDBC with credentials from environment variables:

```properties
# auth-service application.yml
spring.datasource.url=${POSTGRES_URL:jdbc:postgresql://localhost:5433/pawfinder}
spring.datasource.username=${POSTGRES_USER:pawfinder}
spring.datasource.password=${POSTGRES_PASSWORD:pawfinder_dev}
```

In production on Render, these are injected via the Render Dashboard Environment tab or from Render's auto-generated database credentials.

### Services with Database Access

| Service | Has DB Access | Tables Managed |
|---------|---------------|----------------|
| Auth Service | тЬЕ | `users`, `flyway_auth_history` |
| Alert Service | тЬЕ (not deployed) | `pets`, `alerts`, `sightings` |
| Messaging Service | тЬЕ (not deployed) | `conversations`, `messages` |
| Analytics Service | тЬЕ (not deployed) | `daily_metrics` |

> **Note:** The API Gateway, Media, Reward, Matching, and Notification services do not have direct database access. They communicate via HTTP with other services or external APIs (AWS S3, Stripe, AWS Rekognition).

## ЁЯФР Security Notes

- **Password hashing:** BCrypt (via `password_hash` column)
- **Phone hashing:** SHA-256 (via `phone_hash` column)
- **Email hashing:** SHA-256 (via `email_hash` column) тАФ plaintext `email` also stored for login lookup (V7)
- **No direct public access:** Database is internal-only within Render's Oregon network
- **Connection string exposure:** Never in source code тАФ always via environment variables
- **Non-root DB user:** `pawfinder` user (not `postgres` superuser)

## ЁЯз╣ Maintenance

### PostGIS Extension

The PostGIS extension must be enabled manually after database creation:

```sql
CREATE EXTENSION IF NOT EXISTS postgis;
```

This is handled by Flyway migration V1 in the Alert Service (`V1__init.sql` enables `uuid-ossp`; PostGIS is enabled separately on the Render managed database).

### Database Backups

Render automatically creates daily backups for managed PostgreSQL databases (Starter plan). Backup retention: 7 days.
