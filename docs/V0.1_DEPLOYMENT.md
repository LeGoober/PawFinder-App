### Known Issue: Render DNS + Upstash

Render's internal DNS sometimes fails to resolve the Upstash hostname (`tough-kangaroo-68983.upstash.io`). This causes intermittent health check failures on the gateway.

**Mitigation:** `MANAGEMENT_HEALTH_REDIS_ENABLED=false` is set on the gateway to exclude Redis from the actuator health check. The Lettuce Redis client handles transient DNS failures internally with automatic reconnection.

## 🔧 Deploy Method

### `deploy-render.ps1` — Render API Deployment Script

The V0.1 infrastructure is created via the Render REST API, not the Render Blueprint. This gives per-service control and better error visibility.

**Usage:**

```powershell
.\deploy-render.ps1 `
  -ApiKey "rnd_..." `
  -OwnerId "tea_..." `
  -DbHost "dpg-d8r47dj7uimc73elmf60-a.oregon-postgres.render.com" `
  -DbPassword "<password>" `
  -RedisHost "tough-kangaroo-68983.upstash.io" `
  -RedisPort "6379" `
  -JwtSecret "$(openssl rand -hex 32)"
```

**Options:**
- `-Services "pawfinder-auth-service,pawfinder-api-gateway"` — Deploy only specified services
- `-SkipFrontend` — Skip frontend deploy
- `-DryRun` — Show what would be created without creating

**What it does:**
1. Creates all 9 backend web services + frontend via `POST /v1/services`
2. Sets environment variables on each service
3. Configures auto-deploy from GitHub `main` branch
4. Handles 409 (already exists) gracefully

### Alternative: Render Blueprint

The `render.yaml` file defines all services declaratively. Push to GitHub and Render auto-deploys. However, the CLI/API method is preferred for V0.1 due to per-service control.

## 🥶 Cold Start Behavior

Render's **free/starter tier** puts services to sleep after **15 minutes of inactivity**. On the next request:

| Service Type | Wake-up Time |
|-------------|-------------|
| Web Service (Docker) | 30-60 seconds |
| Managed PostgreSQL | Instant (managed services don't sleep) |
| Upstash Redis | Instant (serverless, always warm) |

**Impact on V0.1:**
- First user request after idle period will experience slow loading
- Subsequent requests are fast once the service is warm
- Health checks on Render's side resume immediately after wake-up
- This is expected behavior for free tier and is acceptable for V0.1 demo purposes

## 🔄 Deployment Workflow

```
Developer pushes to main
         │
         ▼
Render detects push (auto-deploy enabled)
         │
         ▼
Render builds Docker image (multi-stage)
  Stage 1: maven:3.9-eclipse-temurin-21-alpine
    ├─ Build pawfinder-shared
    └─ Build target service (mvn package -DskipTests)
  Stage 2: eclipse-temurin:21-jre-alpine
    └─ Copy JAR, create non-root user
         │
         ▼
Render deploys container
         │
         ▼
Health check: GET /actuator/health
  (every 30s, max 3 retries)
         │
         ▼
Service is live ✓
```

## 🐳 Docker Images

All services use a consistent two-stage build pattern:

```dockerfile
# Stage 1: Maven Build
FROM maven:3.9-eclipse-temurin-21-alpine AS build
# ... copy shared + service source, run mvn package ...

# Stage 2: Runtime
FROM eclipse-temurin:21-jre-alpine
COPY --from=build /app/target/*.jar app.jar
RUN addgroup -S pawfinder && adduser -S pawfinder -G pawfinder
USER pawfinder
HEALTHCHECK ... CMD wget -qO- http://localhost:8080/actuator/health || exit 1
ENTRYPOINT ["java", "-jar", "app.jar"]
```

**Security:** Non-root user (`pawfinder:10001`) runs the JVM in all services.

## ⚠️ Known Deployment Issues

### 1. Render DNS + Upstash Resolution

**Symptom:** Gateway health check occasionally fails with Redis connection timeout.  
**Cause:** Render's internal DNS intermittently fails to resolve `tough-kangaroo-68983.upstash.io`.  
**Mitigation:** `MANAGEMENT_HEALTH_REDIS_ENABLED=false` — skips Redis health check entirely.  
**Impact:** None on rate limiting. Lettuce client handles reconnection internally.

### 2. Render Blueprint `--build-arg` Limitation

**Symptom:** Single generic Dockerfile with `ARG SERVICE_NAME` fails on Render.  
**Cause:** Render Blueprint does not support `--build-arg` for Docker builds.  
**Solution:** Created per-service Dockerfiles (`Dockerfile.auth-service`, `Dockerfile.api-gateway`, etc.).  
**Note:** The generic `Dockerfile` still works for local/manual builds.

### 3. Free Tier Sleep

**Symptom:** First request after idle period is very slow.  
**Cause:** Render free tier puts services to sleep after 15 min of inactivity.  
**Mitigation:** None required — this is expected behavior. Consider a keep-alive ping if needed.

### 4. Google Client ID Hardcoded in Frontend

**Symptom:** Changing Google OAuth client ID requires a full rebuild.  
**Cause:** Google Client ID is baked into `web/index.html` at build time.  
**Mitigation:** Rebuild and redeploy the frontend Docker image when rotating credentials.

## 🚨 Disaster Recovery

**If all services go down:**
1. Check Render status: https://status.render.com
2. Verify GitHub repo is accessible
3. Redeploy via Render Dashboard (Manual Deploy → Deploy latest commit)
4. If persistent, run `deploy-render.ps1` against fresh Render project

**If database is lost:**
1. Render managed PostgreSQL includes automated daily backups
2. Contact Render support for restore
3. Flyway migrations re-apply schema automatically on next deploy
